# 설명하는 주석

## 요약

- 가장 좋은 코드는 결합도를 낮춘 코드겠지만, 이것이 당장 불가능할 때에는 주석으로 추가해주세요.

```ts filename="설명하는_주석_예시.tsx"
/**
 * [ 컨텍스트 설명 ]
 *
 * A씨는 foo라는 라이브러리의, run라는 메서드를 사용하려 해요.
 * 하지만, 이를 트리쉐이킹을 의식하여 import하려 했더니, import에러가 발생했어요.
 * 알고보니, esm에서 named export가 호환이 되지 않는 방식으로 사용되는 라이브러리였어요.
 * 따라서 이를 설명하는 주석을 작성하기로 했어요.
 */

/** NOTE 해당 라이브러리는 default export로 구현되었기 때문에 별도의 자동완성을 통해 named export로 호출하면 문제가 발생헙니다. */
import Class from "foo";

export const runClass = () => {
  Class.run();
};
```

## 사유

주석... 이전부터 개발계에서 많이 논해지는 주제인데 말이죠. (먼 산을 바라보며 🌋🙃)

설명하는 주석은 있을 수록 개발적인 삶이 윤택해진다고 보는 편이에요. **특히 유틸 함수에서 말이죠**

모두의 코드가 완벽하면 좋겠지만, 가끔은 결합도가 떨어짐에도 우선적으로 배포를 위해 반영된 공통 함수들이 만들어질 때가 있어요.

이럴 때에는 항상 FIXME 등을 사용해주거나, 혹은 어떤 특정 케이스에서만 당장 사용한지라도 서술해주면 같이 관리하기가 참 좋아요.
