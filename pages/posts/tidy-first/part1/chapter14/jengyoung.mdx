# 설명하는 주석

## 요약

- 가장 좋은 코드는 결합도를 낮춘 코드겠지만, 이것이 당장 불가능할 때에는 주석으로 추가해주세요.

```ts filename="설명하는_주석_예시.tsx"
/**
 * [ 컨텍스트 설명 ]
 *
 * A씨는 현재 클라이언트 관련한 정보들을 별도로 상수화하여 관리하고 있어요.
 * 하지만 이러한 변수명들을 불러올 때, 다음과 같은 고민이 들었어요.
 *
 * - `URL`이라는 변수명은 너무 반복돼요. 무엇인지 확인하려면 한 번 더 들어가야 했어요.
 * - 관련된 코드가 여러 개 있다는 것을 인지시키고 싶었어요. 실제로 현재 도메인 이름이 대상 도메인이 맞는지 체크하는 함수가 존재했는데, 이를 인지하는 사람이 적었어요.
 * - 각각을 가져왔을 때의 이점이 있나 싶었어요. 성능상 크게 차이나지 않을뿐더러, 어차피 각 값과 함수가 한 번씩은 CSR 구조 내에서 한 번씩은 반드시 사용돼요.
 */

class Host {
  static BASE_URL = "https://www.foo.com";

  static checkValidURL(url: string) {
    // ...
  }
}
```

## 사유

주석... 이전부터 개발계에서 많이 논해지는 주제인데 말이죠. (먼 산을 바라보며 🌋🙃)

설명하는 주석은 있을 수록 개발적인 삶이 윤택해진다고 보는 편이에요. **특히 유틸 함수에서 말이죠**

모두의 코드가 완벽하면 좋겠지만, 가끔은 결합도가 떨어짐에도 우선적으로 배포를 위해 반영된 공통 함수들이 만들어질 때가 있어요.

이럴 때에는 항상 FIXME 등을 사용해주거나, 혹은 어떤 특정 케이스에서만 당장 사용한지라도 서술해주면 같이 관리하기가 참 좋아요.

### 논의해보고 싶은 부분 - 여러분의 주석 강박증은 어때요?

저는 주석 강박증이 좀 있는 편이에요.

특히 패키지 관련한 설정에 관해서는 더 심해져요.

다음은 현재 제가 관리하고 있는 앱 부분인데요. 이전에 앱 패키지로 호되게 당하다보니, 다음과 같이 처리하고 있어요.

```ruby
# react-native-config (v1.5.2)
# see: https://www.npmjs.com/package/react-native-config
pod 'react-native-config', :path => '../node_modules/react-native-config'
```

여러분은 어떤 주석에 관한 습관을 갖고 있나요?

여기서 습관이란, **주석을 만들지 않는 것**도 포함이에요!
