# 설명하는 변수

## 요약

- 어떤 복잡한 표현식을 한 눈에 보기 어렵다면, 의도가 보이도록 변수로 표현식을 추출하여 다시 할당하세요.
- 만약 특정 표현식이 일부 변경되어야만 하더라도, 각 식의 의미가 알기 쉽게끔 토큰화되어 서술되어 있기 때문에, 쉽게 파악하고 교체할 수 있어요.

```tsx filename="설명하는_변수_예시.tsx"
/**
 * [ 컨텍스트 설명 ]
 *
 * A씨는 현재 스토리지에서 특정 값을 가져와야 하는 상황입니다.
 * 이때, 스토리지명이 계속해서 반복해서 사용됨에도 불구하고 별도로 값을 재사용하지 않는 것이 마음에 들지 않았습니다.
 *
 * 또한, 특정 키들은 모호하게 키 이름이 적혀 있어 해석하기 난감했습니다.
 * 이러한 점들을 개선하기 위해 설명하는 변수를 만드려 합니다.
 */

const SessionStorageKeys = {
  설명하는변수_예시: "example",
};

const getSessionStorageValue = (
  key: keyof typeof SessionStorageKeys,
  options?: SessionStorageStoreOptions
) => {
  // ...
};

getSessionStorageValue("설명하는변수_예시");
```

## 사유

> 예시를 디테일하게 설명하기보다는 좀 더 사유를 디테일하게 쓰는 게 좋은 것 같아, 이번 모임에서는 사유를 좀 더 적어봤어요.

설명하는 변수는 사내 아이디어 디벨롭할 때에도 자주 언급되었듯, 사소하지만 정말 중요한 파트라고 생각해요.

이 파트가 중요한 이유는, 단순히 개발 실력보다는 커뮤니케이션 능력을 판가름할 수 있다고 생각하기 때문이에요.

이전에 제가 어떤 인원 모집글을 본 적이 있어요.  
그때 이러한 문구를 본 적이 있었는데, 마침 이 파트에 적합한 것 같아 들고 와요.

> 저희는 이러한 사람들을 찾아요!
>
> `const a = 2 * 2 * 2 * 2`가 아닌 `const a = 2 ^ 4`라고 말할 수 있는 사람

물론 이 글의 취지는 이해가 되지만, 깊게 따져보면 과연 이 글의 정답이 있을까요?

물론 후자가 더 깔끔하고 선언적인 것처럼 보이지만, 이 특정 식이 2의 네 제곱이라는 의미보다는 각 특정 지표를 하드코딩한 값이었다면 어떨까요?

그래서 저는 설명하는 변수라는 챕터도 의미 있다고 생각하지만, **설명할 수 있는 표현식**이라는 챕터를 별도로 만들고 발제했다면 더욱 낫지 않았을까 싶어요.

어떻게 변수로 잘 설명할 것인지는 주석이 해결하지만, 결국 이 챕터의 시작은 어떻게 이 표현식을 잘 전달할 수 있을까?이지, 변수가 우선이 아니라고 생각하기 때문이에요.
