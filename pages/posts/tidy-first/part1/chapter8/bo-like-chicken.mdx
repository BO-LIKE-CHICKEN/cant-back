# 설명하는 변수

## 어렵고 크고 복잡한 코드의 표현식을 이해했다면 전체에서 일부 표현식을 추출한 후, 표현식의 의도가 드러나도록 변수 이름을 만들어 할당하자

### 사례

이 코드는 query가 비어있는지 체크하는 것 같지만, 그 의도는 코드만 봐서는 파악하기 어렵습니다. 또한, 비어있는 객체를 비교하는 표현식이 복잡한 의도를 내포하고 있으므로 이를 설명하는 변수로 바꿔보겠습니다.

#### AS-IS

```ts
useEffect(() => {
  if (query === {}) {
    return;
  }

  foo();
}, [query]);
```

#### TO-BE

```ts
useEffect(() => {
  const isInitialPageLoad = Object.keys(query).length === 0;

  if (isInitialPageLoad) {
    return;
  }

  foo();
}, [query]);
```

### 변경으로 인한 개선점

#### 가독성 향상:

`isInitialPageLoad`라는 설명하는 변수를 사용하여 `query`의 상태가 페이지의 첫 진입 여부를 의미한다는 의도를 명확하게 드러냈습니다. 이제 코드를 읽는 사람은 이 조건이 첫 페이지 로드를 확인하는 것임을 쉽게 이해할 수 있습니다.

#### 유지보수성:

코드의 조건이 더 복잡해지거나 다른 조건으로 변경되어야 할 때, `isInitialPageLoad` 변수에 할당된 표현식만 변경하면 되므로 코드의 변경이 수월해집니다.

#### 의미 전달:

조건문의 복잡한 표현식을 변수로 추출하여 그 의도를 설명함으로써 코드를 읽는 사람이 `query`에 대한 체크가 페이지 첫 진입을 감지하는 것임을 명확히 알 수 있습니다.

이렇게 설명하는 변수를 적용하면 표현식과 분리되었기 때문에 다음 번 코드를 변경할 때, 둘 중 하나만 읽으면 되니까 재빨리 변경할 수 있습니다.

## BONUS

다만, 코드 정리에 대한 커밋과 동작 변경에 대한 커밋은 분리해야 한다는 것을 놓치지 않아야 합니다.
